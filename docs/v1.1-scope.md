# Safe UGC UI v1.1 Scope

Status: implemented on main

## Baseline

- Current published package version: `1.0.0`
- Current main-branch baseline: published `1.0.0` plus `v1.1` implementation on `main`

This milestone should improve visual expressiveness without expanding SUU's
state model, evaluator surface, or interaction model. The goal is to add a
small "safe visual/layout pack" that stays declarative, validator-friendly,
and easy for the renderer to map.

## Goals

- Improve visual variety for cards without introducing raw CSS escape hatches
- Add a more useful responsive baseline beyond `compact` only
- Keep all new style features structurally typed and validator-enforced
- Avoid changes to condition evaluation, fragment semantics, or local interactive state
- Preserve SUU's current "closed DSL" security posture

## In Scope

### 1. `responsive.medium`

Add a second responsive breakpoint at the node level.

Example:

```json
{
  "type": "Row",
  "style": { "gap": 16, "alignItems": "center" },
  "responsive": {
    "medium": {
      "flexDirection": "column",
      "alignItems": "start"
    },
    "compact": {
      "gap": 10
    }
  }
}
```

Initial semantics:

- `compact` remains the smallest breakpoint
- `medium` is an additional breakpoint above `compact`
- recommended initial threshold: `<= 768px`
- merge order should be:
  - base `style`
  - `responsive.medium`
  - `responsive.compact`
- only the same safe style subset allowed in `compact` should be allowed in `medium`

Why this is in scope:

- It improves author control for tablet and narrow desktop layouts
- It reuses the existing responsive merge model instead of inventing a new one
- It has low security risk because it does not add arbitrary CSS

### 2. `backdropBlur`

Add a safe, single-purpose blur control instead of opening full `backdropFilter`.

Example:

```json
{
  "type": "Box",
  "style": {
    "backgroundColor": "rgba(15, 23, 42, 0.68)",
    "backdropBlur": 12,
    "borderRadius": 18
  }
}
```

Initial shape:

```ts
type BackdropBlur = number;
```

Rules:

- `backdropBlur` accepts only a number
- interpreted as pixel blur radius
- recommended range: `0..40`
- negative values are rejected
- renderer maps it to `backdrop-filter: blur(...)`
- no raw `backdropFilter` string support

Why this is in scope:

- It enables modern glass-style UI without opening a freeform filter string
- A single numeric control is easy to validate and serialize
- It does not touch state, evaluation, or tree traversal logic

### 3. Structured `clipPath` subset

Add a constrained clip-path model using structured objects only.

Supported subset:

- `circle`
- `ellipse`
- `inset`

Example:

```json
{
  "type": "Image",
  "src": "@assets/avatar.png",
  "style": {
    "width": 120,
    "height": 120,
    "clipPath": {
      "type": "circle",
      "radius": "50%"
    }
  }
}
```

Example:

```json
{
  "type": "Box",
  "style": {
    "clipPath": {
      "type": "inset",
      "top": 0,
      "right": 0,
      "bottom": 0,
      "left": 0,
      "round": 20
    }
  }
}
```

Initial shape:

```ts
type ClipPath =
  | { type: 'circle'; radius: Length }
  | { type: 'ellipse'; rx: Length; ry: Length }
  | {
      type: 'inset';
      top: Length;
      right: Length;
      bottom: Length;
      left: Length;
      round?: Length;
    };
```

Rules:

- `clipPath` must be a structured object, never a raw CSS string
- only the listed shapes are allowed
- `Length` follows the existing safe length model already used in styles
- `polygon()`, `path()`, `xywh()`, `url()`, and arbitrary function strings are out of scope
- renderer maps the structured object to a CSS `clip-path` string internally

Why this is in scope:

- It adds meaningful shape variation for cards and images
- A closed object model avoids string injection and parser ambiguity
- The subset is expressive enough to matter without becoming a general path language

### 4. Validator / renderer parity for new style features

The visual additions above should behave like existing safe style features:
one schema shape, one validator interpretation, one renderer mapping.

Required semantics:

- `responsive.medium` follows the same merge and allowlist model as `compact`
- `backdropBlur` has explicit numeric bounds in validator and runtime mapping
- structured `clipPath` is validated shape-by-shape before render
- no new raw CSS string fields are introduced
- JSON Schema, validator, and renderer stay aligned on accepted inputs

Why this is in scope:

- The main risk in visual expansion is validator/runtime drift
- A small style pack only stays safe if every layer agrees on the same closed shape

## Explicit Non-Goals For v1.1

These ideas may be valid later, but they should not be part of this milestone:

- raw `backdropFilter`
- raw string `clipPath`
- `polygon`, `path`, `url`, or arbitrary custom clip-path functions
- keyframe animation
- arbitrary CSS filters
- additional interactive components such as `Carousel`
- general `$expr`
- fragment parameters or slot composition
- a full responsive system with named desktop/wide breakpoints

Keeping these out of scope preserves the low-risk nature of the milestone.

## Proposed Shape

### Responsive

```ts
type ResponsiveProps = {
  medium?: ResponsiveStyle;
  compact?: ResponsiveStyle;
};
```

Notes:

- `medium` should reuse the current responsive style subset
- `compact` remains supported and should keep existing behavior

### `backdropBlur`

```ts
type Style = {
  backdropBlur?: number | Ref<number>;
};
```

Notes:

- a top-level numeric field is preferred over a generic filter object
- this should remain narrower than `boxShadow` or `transform`

### Structured `clipPath`

```ts
type ClipPath =
  | { type: 'circle'; radius: Length | Ref<Length> }
  | { type: 'ellipse'; rx: Length | Ref<Length>; ry: Length | Ref<Length> }
  | {
      type: 'inset';
      top: Length | Ref<Length>;
      right: Length | Ref<Length>;
      bottom: Length | Ref<Length>;
      left: Length | Ref<Length>;
      round?: Length | Ref<Length>;
    };

type Style = {
  clipPath?: ClipPath;
};
```

Notes:

- leaf-level `$ref` support is acceptable if it follows existing structured style object rules
- top-level `clipPath` itself should stay object-literal only

## Validation Rules

The validator should enforce:

- `responsive.medium` exists only under node-level `responsive`
- `responsive.medium` uses the same safe style subset as `compact`
- `backdropBlur` is numeric and within the allowed range
- `clipPath.type` is one of the approved shapes only
- every `clipPath` shape contains only the allowed fields for that shape
- no raw CSS function strings are accepted as substitutes

Suggested error cases:

- invalid `responsive.medium` field
- `backdropBlur` out of range
- unknown `clipPath.type`
- invalid `clipPath` shape object
- forbidden string `clipPath`

## Package Impact

### `packages/types`

- Extend `responsive` schema and types with `medium`
- Add `backdropBlur` style field
- Add structured `clipPath` style field and supporting shape schemas

### `packages/schema`

- Regenerate JSON Schema for the new responsive and style shapes

### `packages/validator`

- Validate `responsive.medium` using the same responsive style policy as `compact`
- Add bounds checking for `backdropBlur`
- Add structural validation for the supported `clipPath` subset
- Keep security filtering aligned with the new style shapes

### `packages/react`

- Extend responsive handling to evaluate `medium`
- Map `backdropBlur` to safe CSS output
- Map structured `clipPath` objects to CSS strings

### `packages/demo`

- Add one demo card that uses:
  - `responsive.medium`
  - `backdropBlur`
  - at least one structured `clipPath`

## Acceptance Criteria

This milestone is complete when all of these are true:

- cards can use `responsive.medium`
- cards can use `backdropBlur`
- cards can use structured `clipPath` with `circle`, `ellipse`, and `inset`
- JSON Schema, validator, renderer, demo, and current-behavior specs all agree
- `pnpm build` passes
- `pnpm test:run` passes

## Follow-up Candidates After v1.1

If this visual/layout pack lands cleanly, likely next decisions are:

- whether a further visual pack should add animation
- whether `polygon` is worth the added validation complexity
- whether the responsive model should grow beyond `medium` and `compact`
