# Safe UGC UI v1.0 Scope

Status: implemented on main

Implementation order note:

- keep `v1.0` scope as `Accordion + Tabs`
- implement it in `Accordion`-first order
- detailed sequencing lives in [`v1.0-accordion-first.md`](/home/tmdduq96kr/projects/SUU/docs/v1.0-accordion-first.md)

## Baseline

- Current published package version: `1.0.0`
- Current main-branch baseline: published `1.0.0`

This milestone should move SUU from static cards with light actions into
safe, encapsulated interaction. The goal is not to create a general component
framework or client-side scripting language. The goal is to add a very small
set of stateful components whose behavior is fixed by the renderer.

## Goals

- Add first-class interactive containers without opening custom scripting
- Keep all interaction state renderer-owned and local
- Preserve validator and runtime agreement on what hidden content costs
- Expand expressiveness enough for settings panels, FAQs, stat panes, and tabbed dashboards
- Keep the public DSL understandable for authors and predictable for LLM output

## In Scope

### 1. `Tabs`

Add a `Tabs` node with a static list of tab items and renderer-managed selected state.

Example:

```json
{
  "type": "Tabs",
  "defaultTab": "stats",
  "tabs": [
    {
      "id": "stats",
      "label": "Stats",
      "content": {
        "type": "Column",
        "children": [
          { "type": "Text", "content": "HP 72 / 100" },
          { "type": "Text", "content": "Armor 18" }
        ]
      }
    },
    {
      "id": "logs",
      "label": { "$template": ["Logs ", { "$ref": "$logCount" }] },
      "disabled": { "$ref": "$logsLocked" },
      "content": { "$use": "missionLogs" }
    }
  ]
}
```

Rules:

- `tabs` is a static array with at least 1 item and a conservative upper bound
- each tab item has `id`, `label`, and `content`
- `label` uses the same safe text value model as `Button.label`
- `content` accepts a normal node or `$use`
- `defaultTab` is optional and must match one declared tab id if present
- per-tab `disabled` may be a boolean literal or boolean `$ref`
- the renderer owns current selected tab state after initial render

Why this is in scope:

- Tabs cover a large class of compact dashboards and profile/detail layouts
- The state machine is small and deterministic
- The interaction model is familiar and can be implemented accessibly

### 2. `Accordion`

Add an `Accordion` node with a static list of collapsible items and renderer-managed expanded state.

Example:

```json
{
  "type": "Accordion",
  "allowMultiple": false,
  "defaultExpanded": ["profile"],
  "items": [
    {
      "id": "profile",
      "label": "Profile",
      "content": { "$use": "profilePanel" }
    },
    {
      "id": "inventory",
      "label": "Inventory",
      "content": {
        "type": "Column",
        "children": {
          "for": "item",
          "in": "$items",
          "template": { "$use": "inventoryRow" }
        }
      }
    }
  ]
}
```

Rules:

- `items` is a static array with at least 1 item and a conservative upper bound
- each item has `id`, `label`, and `content`
- `allowMultiple` defaults to `false`
- `defaultExpanded` is optional
- when `allowMultiple` is `false`, `defaultExpanded` may contain at most one id
- per-item `disabled` may be a boolean literal or boolean `$ref`
- the renderer owns expanded state after initial render

Why this is in scope:

- Accordion solves FAQ, details, and sectioned inspector layouts cleanly
- It gives meaningful expressiveness without adding arbitrary navigation
- Its state semantics are still simple enough for validator/runtime alignment

### 3. Renderer-Owned Local State Only

Interactive state in `Tabs` and `Accordion` should be local to the renderer.

Required semantics:

- clicking or keyboard interaction updates only local component state
- card `state` is read-only input; SUU does not mutate it
- no author-defined reducers, expressions, or transition logic
- no dynamic creation or deletion of tabs/items at runtime
- no action wiring that can change card structure

Why this is in scope:

- This is the line that preserves SUU as a safe DSL instead of a mini app runtime
- It keeps the attack surface much smaller than general event-driven state

### 4. Hidden Subtree Budgeting

Stateful containers introduce content that may be hidden initially and shown later.
That content still needs to count against existing budgets.

Required semantics:

- all tab panel content counts toward node, text, style, overflow, and loop limits
- all accordion panel content counts toward the same limits, even when collapsed initially
- validator should measure the full interactive envelope, not just the initial visible path
- renderer should either pre-reserve or otherwise enforce the same full-envelope budget

Why this is in scope:

- Without this, interactivity would become a budget bypass
- Hidden content must be treated like deferred visibility, not free content

### 5. Accessibility and Keyboard Behavior

The first stateful milestone should be accessible by default.

Required behavior:

- `Tabs` supports click and keyboard selection
- `Accordion` supports click and keyboard toggle
- selected / expanded state is reflected in semantic attributes
- disabled items are non-interactive and visually distinguishable

Why this is in scope:

- These components are only worth standardizing if their baseline behavior is reliable
- Accessibility is part of the renderer contract, not optional app code

## Explicit Non-Goals For v1.0

These ideas remain valid, but they should not be part of this milestone:

- `Carousel`
- auto-advance or timer-driven UI
- swipe / drag gestures
- persisted local state across rerenders or view switches
- author-defined state machines
- arbitrary event handlers
- `onTabChange` / `onExpand` host callbacks
- fragment parameters or slot composition
- additional expression language features
- arbitrary animation systems for open/close or tab transitions

Keeping these out of scope prevents the first interactive milestone from
turning into a generic client UI runtime.

## Proposed Shape

### `Tabs`

```ts
type TabsItem = {
  id: string;
  label: TextValue;
  content: RenderableNode;
  disabled?: boolean | Ref<boolean>;
};

type TabsNode = BaseNode & {
  type: 'Tabs';
  tabs: TabsItem[];
  defaultTab?: string;
};
```

### `Accordion`

```ts
type AccordionItem = {
  id: string;
  label: TextValue;
  content: RenderableNode;
  disabled?: boolean | Ref<boolean>;
};

type AccordionNode = BaseNode & {
  type: 'Accordion';
  items: AccordionItem[];
  allowMultiple?: boolean;
  defaultExpanded?: string[];
};
```

Notes:

- `content` should reuse the existing `RenderableNode` model so fragments and `$if` keep working
- item arrays are structural and static; they are not `$ref`-driven
- this keeps authoring regular and validation tractable

## Validation Rules

The validator should enforce:

- `Tabs.tabs` and `Accordion.items` are non-empty and under a conservative max count
- item ids are unique within the component
- `defaultTab` refers to a declared tab id
- `defaultExpanded` refers only to declared item ids
- `defaultExpanded.length <= 1` when `allowMultiple === false`
- item `content` is a valid renderable node
- hidden content participates in existing limits

Suggested error cases:

- duplicate item id
- empty item list
- invalid default id
- too many items
- invalid disabled value type

## Package Impact

### `packages/types`

- Add `Tabs` and `Accordion` field schemas
- Add corresponding node schemas and inferred types
- Export item subtypes for docs and downstream typing

### `packages/schema`

- Regenerate JSON Schema so editors and generators can emit `Tabs` and `Accordion`

### `packages/validator`

- Validate item count and id uniqueness
- Validate default state fields
- Traverse item content for node, style, security, and limits
- Count hidden panel content toward the same budgets as visible content

### `packages/react`

- Add `Tabs` and `Accordion` renderer components
- Keep selected / expanded state local to the renderer
- Implement keyboard-accessible interactions
- Enforce runtime budgets consistently with validator semantics

- Update `safe-ugc-ui-card-spec.md`
- Update `safe-ugc-ui-card-spec-lite.md`

- Add one demo card that uses `Tabs`
- Add one demo card that uses `Accordion`

## Acceptance Criteria

- Cards can render `Tabs` and `Accordion`
- Item labels support the same safe text model as existing text-bearing controls
- Tabs and accordion item content can use normal nodes, loops, `$if`, and `$use`
- Hidden panel content counts toward existing validator and runtime budgets
- The renderer owns local interactive state without mutating card `state`
- Keyboard and disabled behavior are covered by tests
- Tests cover duplicate ids, invalid defaults, hidden-content limits, and basic interaction flows

## Deferred Candidate After v1.0

If `Tabs` and `Accordion` land cleanly, the next step can evaluate whether
`Carousel` is worth the added complexity. That should be a separate decision,
because touch gestures, timing, virtualization, and motion policy change the
problem more than the name suggests.
