# Safe UGC UI v1.0 Accordion-First Plan

Status: implemented on main

The follow-up `Tabs` implementation is now also on main, so this document is
kept as the sequencing record that led to the completed `v1.0` milestone.

This document narrows the first implementation slice under
[`v1.0-scope.md`](/home/tmdduq96kr/projects/SUU/docs/v1.0-scope.md).

The product scope for `v1.0` remains:

- `Accordion`
- `Tabs`
- renderer-owned local state
- hidden subtree budgeting
- accessible baseline behavior

But the implementation order should be:

1. common interactive-content foundation
2. `Accordion`
3. `Tabs`

## Why `Accordion` First

`Accordion` is the simpler stateful component in this repo's current architecture.

- State model is smaller: expanded set instead of selected index plus tablist focus rules
- It exercises the hard part first: hidden subtree budgeting
- It uses the same authoring surface we want for `Tabs`: `id`, `label`, `content`, `disabled`
- It validates `$if`, `$use`, and `for...in` inside hidden content without adding tab-selection semantics
- Accessible baseline can be reached with native buttons and controlled regions

In short, `Accordion` is the fastest way to prove that SUU can support safe
stateful containers without breaking validator/runtime alignment.

## Phase A: Common Foundation

Before adding any public node type, build the shared machinery for
"interactive items with nested renderable content".

### A1. Shared item content model

Introduce an internal concept for node fields that contain renderable subtrees
outside normal `children`.

The immediate use cases are:

- `Accordion.items[*].content`
- `Tabs.tabs[*].content`

Design direction:

- keep author-facing JSON explicit; do not invent a generic author DSL like `slots`
- add internal traversal helpers that can enumerate embedded renderable positions
- reuse the existing `RenderableNode` type so `$use` and `$if` continue to work

### A2. Generalized traversal

Current traversal logic mainly understands:

- layout `children` arrays
- `for...in.template`
- fragment expansion

`v1.0` needs traversal to also visit:

- accordion item content
- tab item content

Recommended change:

- add a helper that returns embedded renderable entries for a node
- make validator traversal include those entries in path generation
- keep raw-walk and expanded-walk behavior aligned

Example target paths:

- `views.Main.items[0].content`
- `views.Main.tabs[2].content`

### A3. Shared item validation helpers

Add validator utilities for repeated item-list rules.

These rules should be reusable for both `Accordion` and `Tabs`:

- list is non-empty
- list length is below a conservative max
- item ids are unique
- `disabled` is boolean literal or boolean `$ref`
- `content` is a valid renderable node

Recommended max count:

- `MAX_INTERACTIVE_ITEMS = 16`

That is large enough for real cards and small enough to keep hidden subtree
costs predictable.

### A4. Hidden subtree budgeting

This is the most important shared piece.

Validator rule:

- hidden item content counts exactly like visible content

Runtime rule:

- runtime must reserve budget for the full interactive envelope before
  deciding which panels are actually mounted visibly

Recommended runtime design:

- add a preflight measurement pass before actual render
- walk the selected view and include:
  - layout children
  - loop templates
  - fragments
  - interactive item content
- if the envelope exceeds runtime budgets, fail before partial render

Why this design:

- it keeps runtime semantics close to validator semantics
- it avoids the current visible-only accounting problem for hidden panels
- an extra pass is acceptable under current card size limits

### A5. Shared label resolution

Both `Accordion` and `Tabs` should use the same safe label model as existing
text-bearing controls.

Recommended rule:

- item `label` uses `templatedStringSchema`

This keeps authoring consistent with:

- `Button.label`
- `Badge.label`
- `Chip.label`

## Phase B: `Accordion`

Once Phase A exists, add `Accordion` as the first public interactive node.

### B1. Proposed shape

```ts
type AccordionItem = {
  id: string;
  label: TextValue;
  content: RenderableNode;
  disabled?: boolean | Ref<boolean>;
};

type AccordionNode = BaseNode & {
  type: 'Accordion';
  items: AccordionItem[];
  allowMultiple?: boolean;
  defaultExpanded?: string[];
};
```

### B2. Validator rules

Validator should enforce:

- `items.length >= 1`
- `items.length <= MAX_INTERACTIVE_ITEMS`
- item ids unique within one accordion
- `defaultExpanded` ids all exist
- `defaultExpanded.length <= 1` when `allowMultiple !== true`
- item content traverses through node/style/security/limit validators

Error cases worth naming explicitly:

- duplicate accordion item id
- invalid default expanded id
- too many accordion items
- too many default expanded ids in single-open mode

### B3. Renderer behavior

Renderer should own accordion expansion state locally.

Behavior:

- initial state derives from `defaultExpanded`
- clicking an enabled header toggles that item
- when `allowMultiple !== true`, opening one item closes the others
- disabled items never toggle
- collapsed content is not visibly mounted, but still counted in preflight budget

### B4. Accessibility baseline

Minimum acceptable behavior:

- each accordion header is a native `button`
- `aria-expanded` reflects current state
- `aria-controls` and region ids are wired correctly
- `disabled` maps to real button disabled behavior
- keyboard toggle works through button semantics (`Enter` / `Space`)

Deferred:

- arrow-key roving focus between accordion headers

That can be added later if needed, but it should not block the first slice.

### B5. Demo target

Add one demo card that uses:

- at least 3 accordion items
- one item whose content comes from `$use`
- one item whose content contains a `for...in` loop
- one disabled item

This gives coverage for almost every hard integration point in a single sample.

## Phase C: `Tabs`

`Tabs` should follow only after the shared foundation and `Accordion` are stable.

### Why second

- `Tabs` needs stronger keyboard semantics than `Accordion`
- selected-tab fallback rules are slightly more complex
- focus management matters more for tabs than accordion sections
- once Phase A exists, most of the remaining work is renderer semantics

### Additional work unique to `Tabs`

- `defaultTab` fallback when target is missing or disabled
- selected tab resolution when state changes across rerenders
- tablist semantics
- arrow-key navigation between enabled tabs
- optional `Home` / `End` navigation

## Recommended Implementation Order

### Step 1. Shared types and traversal groundwork

Touch areas:

- `packages/types`
- `packages/validator/src/traverse.ts`
- validator raw-walk helpers
- runtime preflight helper in `packages/react`

Goal:

- make item content traversable and budgetable before any public component is added

### Step 2. `Accordion` types, validator, runtime, tests

Touch areas:

- node and prop schemas
- node validator
- limit validator
- security/style traversal integration
- new React component
- renderer switch case

Goal:

- first safe interactive container lands end-to-end

### Step 3. Demo and spec updates for `Accordion`

Touch areas:

- `packages/demo`
- `safe-ugc-ui-card-spec.md`
- `safe-ugc-ui-card-spec-lite.md`
- `safe-ugc-ui-card-spec.types.ts`

Goal:

- make the new interaction model visible and copyable

### Step 4. `Tabs`

Same package surfaces as Step 2, but reusing the already-proven foundation.

## Exit Criteria For The First Slice

The `Accordion-first` slice is complete when all of these are true:

- item content traversal exists independently of layout `children`
- validator counts hidden accordion content toward limits
- runtime reserves budget for hidden accordion content before rendering
- accordion headers are accessible buttons with correct disabled behavior
- accordion item content supports normal nodes, `$if`, `$use`, and loops
- demo and spec updates exist
- build and full test suite pass

Once that slice is stable, `Tabs` becomes an additive feature rather than a
foundational redesign.
