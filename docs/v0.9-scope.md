# Safe UGC UI v0.9 Scope

Status: implemented on main

## Baseline

- Current published package version: `0.6.0`
- Current main-branch baseline: published `0.6.0` plus implemented `v0.7` and `v0.8` features
- Current main-branch additions since `0.6.0`:
  - node-level `$if`
  - limited condition evaluation
  - `Button` / `Toggle` `disabled`
  - `aspectRatio`
  - structured `$template`
  - `Text.spans`
  - `Text.maxLines` / `truncate`

This milestone should improve authoring ergonomics for larger cards by adding subtree reuse
without opening a general component language, runtime slots, or recursive templates.

## Goals

- Remove repeated JSON subtree copy-paste in larger cards
- Let authors reuse stable layout chunks such as headers, stat rows, and repeated callouts
- Keep reuse purely declarative and static so validator and renderer can agree on expansion
- Make limits and traversal semantics explicit so reuse does not create hidden budget bypasses

## In Scope

### 1. Top-level `fragments`

Add an optional top-level `fragments` object to the card.

Example:

```json
{
  "fragments": {
    "profileHeader": {
      "type": "Row",
      "style": { "gap": 12, "alignItems": "center" },
      "children": [
        { "type": "Avatar", "src": "@assets/avatar.png", "size": 48 },
        {
          "type": "Column",
          "children": [
            { "type": "Text", "content": { "$ref": "$username" } },
            { "type": "Text", "content": { "$ref": "$title" } }
          ]
        }
      ]
    }
  }
}
```

Rules:

- `fragments` is a map of fragment name to a single node subtree
- fragment names use the same identifier pattern as `styles`
- each fragment value must be a valid node subtree
- fragments are local to the card only

Why this is in scope:

- It addresses the largest authoring pain left after `v0.8`
- It does not require runtime code execution
- It keeps reuse explicit and serializable in JSON

### 2. `$use` fragment references

Allow a node position to reference a previously-declared fragment by name.

Example:

```json
{
  "type": "Column",
  "children": [{ "$use": "profileHeader" }, { "$use": "profileStats" }]
}
```

Initial shape:

```json
{ "$use": "fragmentName" }
```

Optional wrapper condition:

```json
{ "$use": "vipBanner", "$if": { "$ref": "$isVip" } }
```

Rules:

- `$use` is allowed anywhere a node is allowed, including `views.*.children[*]` and `for...in.template`
- `$use` names are static strings only
- `$use` does not accept inline prop overrides
- `$use` does not accept `style`, `responsive`, or `children`
- `$if` on the `$use` wrapper is evaluated before fragment expansion

Why this is in scope:

- It gives authors reuse without needing fragment parameters or inheritance
- It keeps expansion deterministic
- It avoids ambiguous merge behavior in the first iteration

### 3. Non-recursive fragments only

The first fragment milestone should stay intentionally conservative.

Explicit constraints:

- fragment definitions may not contain another `$use`
- a `$use` target must resolve directly to a normal node subtree
- no recursion
- no fragment-to-fragment composition

Why this is in scope:

- It keeps traversal and limits implementation tractable
- It prevents cycle handling from becoming the real milestone
- It still solves most copy-paste problems in current cards

### 4. Expansion-aware traversal and limits

Fragments must behave like inline-substituted nodes for all validator and renderer checks.

Required semantics:

- expanded fragment nodes count toward total node count
- expanded `Text` content counts toward text byte limits
- expanded styles count toward style byte limits
- expanded `overflow:auto` nodes count toward overflow limits
- expanded subtree structure participates in stack nesting and loop nesting checks
- renderer runtime limits must match validator limits as closely as possible

Why this is in scope:

- Without this, fragments would become a hidden escape hatch around current budgets
- Reuse is only safe if expansion has the same cost model as copy-pasted nodes

## Explicit Non-Goals For v0.9

These ideas remain valid, but they should not be part of this milestone:

- fragment parameters
- fragment prop overrides
- slot-style composition
- dynamic `$use` names via `$ref`
- fragment references inside fragment definitions
- cross-card imports
- additional responsive breakpoints
- visual features such as `backdropBlur`, structured `clipPath`, or keyframe animation
- stateful components such as `Accordion`, `Tabs`, or `Carousel`

Keeping these out of scope prevents a simple reuse feature from turning into a component system.

## Proposed Shape

### Card-level fragments

```ts
type FragmentMap = Record<string, UGCNode>;

type UGCCard = {
  meta: Meta;
  assets?: AssetMap;
  state?: Record<string, unknown>;
  styles?: Record<string, Style>;
  fragments?: FragmentMap;
  views: Record<string, UGCNode>;
};
```

### `$use` node

```ts
type FragmentUseNode = {
  $use: string;
  $if?: Condition;
};
```

### Children / template positions

Any location that currently accepts `UGCNode` should accept:

```ts
UGCNode | FragmentUseNode;
```

## Validation Rules

The validator should enforce:

- fragment names match the allowed identifier pattern
- `$use` names reference an existing fragment
- fragment definitions do not themselves contain `$use`
- `$use` wrapper shape contains only allowed fields
- fragment expansion is included in limit accounting

Suggested error cases:

- unknown fragment name
- `$use` inside `fragments.*`
- invalid fragment name
- invalid fragment wrapper fields

## Package Impact

### `packages/types`

- Add `fragments` to card schema and types
- Add `$use` node/reference schema
- Extend child/template positions to accept fragment references
- Export fragment-related inferred types

### `packages/schema`

- Regenerate JSON Schema so editors can suggest `fragments` and `$use`

### `packages/validator`

- Validate fragment name format
- Validate fragment existence for each `$use`
- Reject nested `$use` inside fragment definitions
- Validate fragment definitions directly, and traverse expanded fragments where usage context matters
- Keep expansion semantics aligned across validator phases

### `packages/react`

- Resolve `$use` before normal node rendering
- Respect `$if` on the `$use` wrapper before expansion
- Apply runtime budgets to expanded fragment subtrees
- Surface clear runtime errors for missing fragment names

### Specs and LLM-facing docs

- Update `safe-ugc-ui-card-spec.md`
- Update `safe-ugc-ui-card-spec-lite.md`
- Update `safe-ugc-ui-card-spec.types.ts`
- Add one demo card that reuses at least two fragments

## Acceptance Criteria

- Cards can define reusable `fragments`
- Views and loop templates can reference fragments with `$use`
- Fragment references are static and non-recursive
- Validator and renderer agree on expansion behavior
- Expansion counts fully toward existing resource limits
- Tests cover valid reuse, missing fragments, forbidden nested `$use`, and limit accounting

## Follow-up Candidate For v1.0

If fragment reuse lands cleanly, the next milestone can evaluate richer interaction and
component-like abstractions:

- parameterized fragments or slots
- additional responsive modes
- stateful components such as `Accordion`, `Tabs`, and `Carousel`

Those should only be considered after plain subtree reuse proves stable.
