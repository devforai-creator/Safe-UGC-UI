# Safe UGC UI v0.8 Scope

Status: implemented on main

## Baseline

- Current published package version: `0.6.0`
- Current main-branch baseline: published `0.6.0` plus implemented `v0.7` features
- Current main-branch additions since `0.6.0`: node-level `$if`, limited condition evaluation, `Button` / `Toggle` `disabled`, and `aspectRatio`

This milestone should improve authoring ergonomics for text-heavy cards without expanding control
flow or adding a general computation language. The scope stays centered on string composition and
the `Text` component family.

## Goals

- Let card authors compose readable strings without host-side preformatting for every label
- Let a single `Text` node render mixed emphasis without splitting one sentence into many nodes
- Add basic line clamping so dense cards remain legible without manual truncation in host state
- Keep the milestone cohesive: text authoring only, not tree reuse or new visual effects

## In Scope

### 1. Structured `$template`

Add a structured template value for safe string composition.

Example:

```json
{
  "$template": [
    "@",
    { "$ref": "$username" },
    " · Lv.",
    { "$ref": "$level" }
  ]
}
```

Why structured parts instead of placeholder parsing:

- No second mini-language or interpolation parser
- Clear Zod schema and JSON Schema output
- Predictable escaping and easier runtime evaluation
- Better fit for LLM generation than embedded placeholder syntax

Allowed part types:

- string literal
- number literal
- boolean literal
- `null`
- `$ref`

Output semantics:

- Each part resolves independently
- `undefined` parts become empty strings
- numbers, booleans, and `null` stringify with normal JavaScript string conversion
- final output is a single string

Initial allowed fields:

- `Text.content`
- `Text.spans[*].text`
- `Badge.label`
- `Chip.label`
- `Button.label`

Explicit non-goals for `$template`:

- arithmetic
- conditionals
- function calls
- date/number localization helpers
- rich formatting directives

### 2. `Text.spans`

Allow `Text` to render multiple inline spans with different inline styles.

Example:

```json
{
  "type": "Text",
  "spans": [
    {
      "text": "HP ",
      "style": { "color": "#94a3b8" }
    },
    {
      "text": {
        "$template": [
          { "$ref": "$hp" },
          "/",
          { "$ref": "$maxHp" }
        ]
      },
      "style": { "color": "#22c55e", "fontWeight": "bold" }
    }
  ]
}
```

Rules:

- `Text` must use exactly one of `content` or `spans`
- `spans` is an ordered array
- each span must include `text`
- span text accepts literal string, `$ref`, or `$template`
- span styles are inline-only and use a restricted typography subset

Initial allowed span style properties:

- `color`
- `backgroundColor`
- `fontFamily`
- `fontSize`
- `fontWeight`
- `fontStyle`
- `textDecoration`
- `letterSpacing`
- `textShadow`

Explicit restrictions:

- no layout props
- no positioning props
- no `hoverStyle`
- no `transition`
- no `responsive`
- no `$style`
- no nested spans

Suggested safety limits:

- max 32 spans per `Text` node

### 3. `Text.maxLines` and `truncate`

Add first-class text overflow handling to `Text`.

Example:

```json
{
  "type": "Text",
  "content": { "$ref": "$description" },
  "maxLines": 2,
  "truncate": "ellipsis"
}
```

Rules:

- `maxLines` is a positive integer
- `truncate` is optional and may be `"ellipsis"` or `"clip"`
- if `maxLines` is omitted, truncation behavior is inactive
- clamping applies to both `content` and `spans`

Suggested safety limits:

- `maxLines`: 1–10

Renderer behavior:

- single-line clamp uses standard overflow rules
- multi-line clamp uses a predictable CSS line-clamp pattern
- if the browser lacks support for multi-line clamp, degraded rendering is acceptable as long as it does not break safety

## Explicit Non-Goals For v0.8

These stay out of scope for this milestone:

- general `$expr`
- value-level `$cond`
- new condition operators beyond the current `$if` system
- `fragments` / `$use`
- additional responsive breakpoints
- `backdropBlur`
- structured `clipPath`
- structured keyframe animation
- new stateful components such as `Accordion`, `Tabs`, or `Carousel`

This keeps the milestone centered on text authoring rather than mixing in subtree reuse,
layout expansion, or richer visual effects.

## Proposed Shape

### New template type

```ts
type TemplatePart = string | number | boolean | null | Ref<unknown>;
type TemplateValue = { $template: TemplatePart[] };
```

### Text fields

```ts
type TextValue = string | Ref<string> | TemplateValue;

type TextSpan = {
  text: TextValue;
  style?: TextSpanStyle;
};

type TextNode = {
  type: 'Text';
  content?: TextValue;
  spans?: TextSpan[];
  maxLines?: number;
  truncate?: 'ellipsis' | 'clip';
  style?: Style;
  responsive?: ResponsiveProps;
};
```

Validation rule:

- exactly one of `content` or `spans`

### Badge / Chip / Button labels

```ts
label: string | Ref<string> | TemplateValue
```

Security-sensitive fields remain unchanged:

- `Image.src`
- `Avatar.src`
- `Button.action`
- `Toggle.onToggle`

## Package Impact

### `packages/types`

- Add `TemplateValue` schema and types
- Extend selected text-like string fields to accept `$template`
- Extend `Text` props with `spans`, `maxLines`, and `truncate`
- Add `TextSpan` and `TextSpanStyle` schemas
- Enforce the `content xor spans` shape at the node schema level if feasible, otherwise in validator

### `packages/schema`

- Regenerate JSON Schema so editors can suggest `$template`, `spans`, `maxLines`, and `truncate`

### `packages/validator`

- Validate `$template` parts and per-field allowlists
- Validate `Text` uses exactly one of `content` or `spans`
- Validate allowed span style subset
- Validate span count limit
- Validate `maxLines` range and `truncate` enum
- Keep current security checks on `$ref` paths and CSS function restrictions

### `packages/react`

- Add a template resolver that produces strings from structured parts
- Update `Text` rendering to support `spans`
- Apply line-clamp styles for `maxLines` and `truncate`
- Reuse the template resolver for `Badge`, `Chip`, and `Button` labels

### `packages/validator` limits

- Count literal text bytes inside `$template` parts
- Count literal text bytes inside `Text.spans`
- Ensure text-heavy cards still obey global text budget limits

### Specs and LLM-facing docs

- Update `safe-ugc-ui-card-spec.md`
- Update `safe-ugc-ui-card-spec-lite.md`
- Update `safe-ugc-ui-card-spec.types.ts`
- Add at least one demo card that uses `$template`, `spans`, and text clamping together

## Acceptance Criteria

- Card authors can build readable labels without host-precomputed joined strings
- A single `Text` node can show mixed emphasis with `spans`
- `Text` can clamp to a bounded number of lines
- Validator and renderer agree on all new shapes
- Security-sensitive fields remain outside `$template`
- Existing `$if`, `disabled`, and style validation behavior stays unchanged
- Tests cover valid templates, invalid template part types, `content xor spans`, invalid span styles, and text clamping behavior

## Follow-up Candidate For v0.9

Once text authoring is stable, the next milestone should handle subtree reuse rather than more
text features:

- `fragments`
- `$use`
- traversal and limit accounting for reusable subtrees

That keeps tree reuse isolated instead of coupling it to the text rendering changes above.
