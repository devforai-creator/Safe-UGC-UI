# Safe UGC UI v0.7 Scope

Status: implemented on main

## Baseline

- Current published package version: `0.6.0`
- Current implemented baseline: Phase 2 plus `responsive.compact`
- Current security model: whitelist-based schema, validator-enforced safety rules, and renderer-side isolation

This document narrows the next feature milestone into a small, implementation-ready scope.
The goal is to increase practical card expressiveness without reopening general expression
evaluation or raw browser escape hatches.

## Goals

- Add basic conditional rendering so cards can branch on state without host-side view duplication
- Expose already-supported interaction behavior that exists in the renderer but not yet in the schema
- Add one low-risk layout primitive that improves responsive composition
- Keep the milestone small enough to ship with full type, validator, renderer, test, and spec coverage

## In Scope

### 1. Node-level `$if`

Allow any node to declare an optional `$if` field.

Example:

```json
{
  "type": "Badge",
  "$if": { "$ref": "$user.isVip" },
  "label": "VIP",
  "color": "#ffcc00"
}
```

Semantics:

- If `$if` resolves to `true`, render the node normally
- If `$if` resolves to `false`, `null`, or `undefined`, skip rendering the node
- `$if` applies before child traversal and before component-specific rendering
- `$if` is allowed on all node types, including loop templates

Why this is in scope:

- It removes the most obvious expressiveness gap in the current DSL
- It does not require general-purpose computation
- It keeps control flow declarative and easy to validate

### 2. Limited condition evaluation

`$if` should not accept arbitrary JavaScript or a general `$expr`.
Instead, it should accept a small declarative condition AST.

Initial allowed condition forms:

- `true` / `false`
- `{ "$ref": "$path.to.boolean" }`
- `{ "op": "not", "value": <condition> }`
- `{ "op": "and", "values": [<condition>, ...] }`
- `{ "op": "or", "values": [<condition>, ...] }`
- `{ "op": "eq" | "ne" | "gt" | "gte" | "lt" | "lte", "left": <operand>, "right": <operand> }`

Initial operand forms:

- string literal
- number literal
- boolean literal
- `null`
- `$ref`

Explicit constraints:

- No arithmetic operators
- No string concatenation
- No function calls
- No user-authored code strings
- No recursion beyond a small fixed depth

Why this is in scope:

- It is enough for show/hide logic and simple status-based branching
- It avoids reintroducing the removed general `$expr` surface
- It keeps `SECURITY.md` aligned with its current non-goal around general runtime expression evaluation

### 3. `disabled` on `Button` and `Toggle`

Expose the `disabled` prop at the card schema/type level for `Button` and `Toggle`.
The React components already support it.

Example:

```json
{
  "type": "Button",
  "label": "Claim Reward",
  "action": "claimReward",
  "disabled": { "$ref": "$reward.claimed" }
}
```

Rules:

- Type: `boolean | $ref`
- `Button.action` and `Toggle.onToggle` remain static strings
- Disabled state affects interactivity only; it does not change event routing semantics

Why this is in scope:

- High usefulness
- Low implementation risk
- Already partially implemented in the renderer

### 4. `aspectRatio` style property

Add `aspectRatio` to the style whitelist.

Example:

```json
{
  "type": "Image",
  "src": "@assets/cover.png",
  "style": {
    "width": "100%",
    "aspectRatio": "16 / 9",
    "objectFit": "cover"
  }
}
```

Allowed value forms:

- number
- ratio string such as `"16 / 9"`
- optionally `$ref` if it resolves to one of the allowed literal forms

Rules:

- No CSS functions
- No arbitrary unit strings beyond the supported ratio syntax
- Validation should reject malformed ratio strings rather than relying on browser fallback

Why this is in scope:

- It is a meaningful layout improvement for cards and media blocks
- It is low risk compared to more open-ended visual features
- It fits the existing style whitelist model

## Explicit Non-Goals For v0.7

These ideas remain valid, but they should not be part of this milestone:

- General `$expr`
- Value-level `$cond`
- `$template` string interpolation
- `Text.spans`, `maxLines`, or truncate behavior
- `fragments` / `$use`
- Additional responsive breakpoints beyond `compact`
- `backdropBlur`
- structured `clipPath`
- structured keyframe animation
- new stateful components such as `Accordion`, `Tabs`, or `Carousel`

Keeping these out of scope prevents the milestone from mixing control flow, computed values,
text formatting, and advanced visuals into one release.

## Proposed Shape

### Node schema

Add to the shared base node fields:

```ts
$if?: Condition
```

### Interaction props

Add to `Button` and `Toggle` fields:

```ts
disabled?: boolean | Ref<boolean>
```

### Style schema

Add to core style props:

```ts
aspectRatio?: number | string | Ref<number | string>
```

The validator should enforce the accepted literal formats.

## Package Impact

### `packages/types`

- Add condition types and Zod schemas
- Extend base node fields with `$if`
- Add `disabled` to button/toggle props
- Add `aspectRatio` to style props and exported types

### `packages/schema`

- Regenerate JSON Schema so editors see `$if`, `disabled`, and `aspectRatio`

### `packages/validator`

- Validate condition AST shape and depth
- Validate allowed operand types for comparisons
- Validate `disabled` as boolean or `$ref`
- Validate `aspectRatio` literal formats
- Ensure existing security checks still apply to any `$ref` paths used inside `$if`

### `packages/react`

- Add a small condition resolver separate from general value resolution
- Skip node rendering when `$if` is falsy
- Pass resolved `disabled` through to `Button` and `Toggle`
- Map `aspectRatio` into React CSSProperties

### Specs and LLM-facing docs

- Update `safe-ugc-ui-card-spec.md`
- Update `safe-ugc-ui-card-spec-lite.md`
- Update `safe-ugc-ui-card-spec.types.ts`

## Acceptance Criteria

- A card can show or hide nodes with `$if`
- `$if` supports boolean refs and simple comparisons without any general evaluator
- `Button` and `Toggle` accept `disabled`
- `aspectRatio` validates and renders correctly
- Validator and renderer behavior match for all new forms
- Security invariants for assets, CSS functions, position rules, and limits remain intact
- Tests cover happy path, invalid AST shapes, invalid comparison operands, malformed ratios, and renderer behavior

## Follow-up Candidate For v0.8

If v0.7 lands cleanly, the next milestone should focus on authoring ergonomics rather than
more control flow:

- `$template`
- text spans and truncation
- fragments / `$use`

That keeps the next step incremental instead of immediately expanding into a full expression
language.
